# Slides — Case Studies Scenario

**Implements:** [describe a case study](../../../patterns/help-users/describe-a-case-study.md) — the shared cross-medium concept, where the user goal and medium-agnostic principles live. This file owns the *how* in slides.

**Job-to-be-done:** Tell a specific client story with credibility. Stands alone as a sales artefact, also slots into bigger decks.

**Typical deck shape:** Single slide (summary) OR 2–3 slides (challenge / approach / outcomes). The case-study layout supports both.

**See also:** [layouts.md](../layouts.md) (*Case study slide*).

---

## Do

- Use the canonical case-study layout (diagonal ribbon, client logo top-left, two-panel right).
- Open with the client's challenge framed in their words, not EE's.
- Lead the outcomes panel with bolded headline metrics, then qualitative outcomes.
- Name the technologies, scale, and team shape where it adds credibility ("at a scale of 40 teams, 70 services, 150 microservices on GCP").
- Get client approval for the named case study before external use — flag if approval status is unknown.

---

## Don't

- Do not anonymise where the client has approved naming — named cases carry more weight.
- Do not anonymise to the point of meaninglessness ("a large UK retailer" + no metrics = no value). Either name it or use it as illustrative content elsewhere.
- Do not over-claim. "Helped reduce" not "transformed" unless it really was a transformation.
- Do not include team size + day rate inferences — that's commercial info, not case-study info.
